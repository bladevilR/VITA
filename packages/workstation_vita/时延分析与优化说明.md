# 时延分析与优化说明

## 一、结论

- 原来的主要时延，不在 embedding，也不在统计接口，而是在**诊断问答链路**。
- 诊断问答原来主要慢在两块：
  1. 多层 `support` 对服务端串行调用
  2. 大模型规划和最终生成
- 这次优化后，第二轮 50 条真实问法回归结果：
  - 通过率：`50/50`
  - 平均耗时：`33.65 秒 -> 16.28 秒`
  - 最长耗时：`115.91 秒 -> 45.91 秒`

## 二、原始瓶颈在哪

### 1. 诊断 support 串行调用

原来一个典型诊断问题，工作站会按层依次请求服务端：

- 直接匹配
- 本站同设备历史
- 本站同专业历史
- 同类设备同现象
- 相近现象参考
- 同专业补充参考

以真实样例为例：

- `P13 胥江路站TVM5重启后还是卡死怎么处理`
  - 优化前总耗时：`60.894 秒`
  - 其中 support 串行总耗时：`33.95 秒`
  - 占比约：`55.8%`

- `P24 石路站扶梯停止后先查什么`
  - 优化前总耗时：`58.741 秒`
  - 其中 support 串行总耗时：`33.544 秒`
  - 占比约：`57.1%`

### 2. 大模型调用过多

诊断问答原来经常至少要走：

- 解析模型
- 检索规划模型
- 最终回答模型

其中最终回答模型最慢，规划模型次之，解析模型在本地规则没覆盖到真实问法时也会额外多付一次。

真实样例：

- `P13`
  - 优化前 LLM 总耗时：`25.845 秒`
- `P24`
  - 优化前 LLM 总耗时：`24.413 秒`
- `P08 陆慕站综合监控工作站历史工单帮我整理`
  - 优化前 LLM 总耗时：`7.938 秒`

### 3. embedding 不是主要瓶颈

真实样例里 embedding 基本都在：

- `0.05 ~ 0.08 秒 / 次`

即使 3 次合计也只有 `0.2 秒` 左右，基本可以忽略。

### 4. 统计、责任归属本来就不慢

真实样例：

- `P35 2号线近7天哪个站故障最多`：`0.347 秒`
- `P46 胥江路站TVM5卡死一般联系谁`：`0.516 秒`

所以真正需要优化的是**诊断类**，不是全系统所有功能。

## 三、这次已经落地的优化

### 1. 本地解析覆盖更多真实口语

补了这类关键词：

- `先查`
- `先看`
- `先检查`
- `查什么`
- `查哪里`
- `看什么`
- `看哪里`
- `优先排查`
- `怎么弄`

作用：

- 减少本来可以本地解析却又去调一次解析模型的情况。

### 2. 精确场景跳过检索规划模型

对以下场景直接用启发式检索计划，不再额外调一次规划模型：

- `车站 + 设备/专业 + 明确故障现象`
- `历史查询 + 明确车站 + 明确设备/专业`

作用：

- 省掉一次 2 到 4 秒左右的模型调用。

### 3. support fallback 从串行改为并发

在服务端没有批量接口时，工作站侧现在会并发跑多层 support。

当前默认并发度：

- `VITA_SUPPORT_PARALLELISM=3`

作用：

- 把原来 6 层依次排队，改成 3 路并发，直接压掉最大头的等待时间。

### 4. 模型调用关闭 thinking，并收紧输出 token

这次把模型调用改成了：

- `enable_thinking=false`
- JSON 任务 token 更小
- 最终回答 token 上限收紧到 `4096`

作用：

- 减少无意义推理时间
- 减少超长输出带来的尾部耗时

### 5. 修正历史证据被覆盖的问题

在提速过程中出现过一次回归：

- 同一个工单同时出现在“直接层”和“本站历史层”时，原来的汇总去重会把“本站历史”那条证据挤掉。

现在已经修复：

- 保留 `station_history` 版本，不再被同 ticket 的其他层覆盖。

## 五、现在剩下的主要时延在哪

优化后，新的第一瓶颈已经变成：

- **最终回答模型生成**

从样例看：

- `P13` 优化后总耗时 `34.198 秒`
  - embedding 合计约 `0.205 秒`
  - 最终回答模型约 `21.322 秒`
  - 剩余约 `12.7 秒` 基本就是并发后的 support 检索耗时

- `P24` 优化后总耗时 `30.586 秒`
  - embedding 合计约 `0.205 秒`
  - 最终回答模型约 `18.345 秒`
  - 剩余约 `12.0 秒` 主要也是 support

所以当前排序已经变成：

1. 最终回答模型
2. 并发后的 support 检索
3. 其他环节基本都比较小

## 六、下一步还可以怎么继续压

### 1. 最值得做：缩短最终回答模型的负担

可以继续做：

- 缩短 prompt 里重复的统计和案例文本
- 控制最终回答字数，不要让模型展开太长
- 对高置信场景直接走“模板化回答 + 少量模型润色”

这是下一步最值的，因为它已经是新的第一瓶颈。

### 2. 服务端真正上批量 support 接口

现在虽然工作站侧已经并发了，但还是多次 HTTP 请求。

如果服务端真正支持：

- 一次请求带多层条件
- 一次返回多层结果

还能继续再降一截。

### 3. 做短时缓存

适合缓存的对象：

- 相同问题的最终回答
- 常见问题的检索计划
- 相同 search query 的向量候选

典型收益场景：

- 车站值班员连续多次问同一个故障
- 多个用户短时间内问同类高频故障

### 4. 高置信场景做“半模板化”

像这几类高频场景：

- 陆慕站综合监控工作站黑屏/无显示
- 胥江路站 TVM5 卡死/重启无效
- 石路站扶梯停止/失电停止

可以考虑：

- 检索仍然保留
- 回答部分改成模板拼装
- 只有复杂补充说明再调用模型

这样会继续降很多时延，而且稳定性更高。
