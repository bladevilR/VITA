## 二、优化内容

### 2.1 优化项 #1: 降低 LLM max_tokens

**位置**: `vita.py` 行 2476

**修改前:**
```python
yield from call_llm_stream(prompt, temperature=0.3, max_tokens=2000)
```

**修改后:**
```python
yield from call_llm_stream(prompt, temperature=0.3, max_tokens=1200)
```

**原因分析:**
- Prompt 要求输出 200-500 字（约 300-750 tokens）
- max_tokens=2000 过大，导致 LLM 预留过多空间
- 实际输出很少超过 1000 tokens

**预期效果:**
- LLM 生成时间减少 35%
- 从 10.0秒 → 6.5秒

---

### 2.2 优化项 #2: 降低 FAISS 检索数量

**位置**: `vita.py` 行 1970

**修改前:**
```python
distances, indices = faiss_index.search(
    np.array([query_embedding], dtype='float32'), k=100  # 从50提升到100
)
```

**修改后:**
```python
distances, indices = faiss_index.search(
    np.array([query_embedding], dtype='float32'), k=50  # 优化：降低到50提升速度
)
```

**原因分析:**
- k=100 检索过多候选，增加计算和数据库查询开销
- 后续有 Rerank 精排，k=50 足够保证质量
- 减少数据库查询数量

**预期效果:**
- 检索时间减少 24%
- 从 3.8秒 → 2.9秒

---

### 2.3 优化项 #3: 降低 Rerank 候选数量

**位置**: `vita.py` 行 2554

**修改前:**
```python
analysis_df, rerank_applied = apply_rerank_to_df(
    df=curated_df.head(50),
    query=user_query,
    top_k=20
)
```

**修改后:**
```python
analysis_df, rerank_applied = apply_rerank_to_df(
    df=curated_df.head(30),  # 优化：从50降到30提升速度
    query=user_query,
    top_k=20
)
```

**原因分析:**
- Rerank API 处理 50 条候选耗时较长
- 30 条候选足够保证排序质量
- 减少 API 调用开销

**预期效果:**
- Rerank 时间减少 41%
- 从 2.2秒 → 1.3秒

## 五、风险评估

### 5.1 降低 max_tokens 的风险

**风险**: 可能导致回答被截断

**缓解措施**:
- Prompt 已明确要求 200-500 字
- 1200 tokens 足够容纳 600-800 字
- 实际测试中未发现截断问题

**监控指标**:
- 检查日志中是否有 "finish_reason": "length"
- 用户反馈回答是否完整

**风险等级**: 🟢 低

---

### 5.2 减少检索数量的风险

**风险**: 可能遗漏相关案例

**缓解措施**:
- k=50 仍然是较大的候选集
- 后续有 Rerank 精排保证质量
- 混合检索（向量+关键词）提供冗余

**监控指标**:
- 对比优化前后的案例相关性
- 用户反馈案例质量

**风险等级**: 🟢 低

---

### 5.3 减少 Rerank 候选的风险

**风险**: 可能影响排序质量

**缓解措施**:
- 30 条候选足够 Rerank 模型精排
- 前置的相关度打分已过滤低质量案例
- 最终只展示 Top 20，30 条候选充足

**监控指标**:
- 对比优化前后的排序质量
- 用户点击率和满意度

**风险等级**: 🟢 低
